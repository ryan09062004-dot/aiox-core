## Story
**As a** usuário do ShapeAI,
**I want** poder alternar entre exercícios de academia e exercícios para casa em meu plano de treino,
**so that** eu não perca um treino quando não puder ir à academia, com alternativas equivalentes que trabalham os mesmos grupos musculares.

## Acceptance Criteria

1. **Alternativas geradas no ai-engine** — O `plan_generator.py` é modificado para incluir um campo `home_alternative` em cada exercício. Cada `home_alternative` tem os mesmos campos que `Exercise` (`name`, `muscle_group`, `sets`, `reps`, `rest_seconds`, `note`), mas usa equipamentos de casa (peso corporal, garrafa d'água, cadeira, elástico, etc.).

2. **Geração simultânea** — As alternativas são geradas no mesmo prompt/chamada da IA que gera o plano principal, não em chamada separada. O JSON retornado tem `home_alternative` inline em cada exercise.

3. **Tipos atualizados** — A interface `Exercise` em `packages/shared/src/types/analysis.types.ts` recebe campo opcional `home_alternative?: HomeExercise | null`. Nova interface `HomeExercise` com mesmos campos de `Exercise` (sem o campo `home_alternative`).

4. **Toggle Academia/Casa** — A tela de treino `workout.tsx` exibe um toggle visual no topo do conteúdo (abaixo do header, acima das semanas). Toggle tem duas opções: "Academia" e "Casa". Default: "Academia".

5. **Troca de exercícios** — Quando o toggle está em "Casa", todos os exercícios de todos os dias exibem o `home_alternative` no lugar do exercício original. O nome, séries, reps e descanso refletem a alternativa. Quando volta para "Academia", exibe o original.

6. **Persistência do toggle** — A preferência do toggle é salva em `AsyncStorage` com chave `workout_mode_${analysisId}`. Ao voltar para a tela, o toggle é restaurado.

7. **Fallback gracioso** — Se um exercício não tiver `home_alternative` (dados antigos antes da feature), o exercício original é exibido em ambos os modos sem quebrar a UI.

8. **Compatibilidade retroativa** — A migration `010_workout_home_alternatives.sql` é uma no-op (só `SELECT 1`) pois as alternativas são armazenadas dentro do JSONB `weeks` da tabela `workout_plans` já existente. Não há mudança de schema.

9. **Visualmente consistente** — O toggle usa o design system do app (dark theme, cor primária `#4CAF50` para item selecionado, fundo `#1a1a1a`, texto `#fff`). Não deve parecer um botão de sistema padrão.

10. **Freemium (mesmo da análise)** — A feature de alternativas é entregue junto com o plano de treino normal, sem nova gate de freemium. Quem tem acesso ao plano de treino tem acesso às alternativas.

## Dev Notes

### Contexto do projeto

- **Tela de treino:** `apps/mobile/app/(app)/analysis/[id]/workout.tsx` — usa `WorkoutDayCard` de `apps/mobile/src/components/workout/WorkoutDayCard.tsx`
- **Plan generator:** `services/ai-engine/app/pipeline/plan_generator.py` — usa Anthropic Claude com fallback manual `_EXERCISES_BY_GROUP`
- **Tipos:** `packages/shared/src/types/analysis.types.ts` — `Exercise`, `WorkoutSession`, `WorkoutWeek`, `WorkoutPlan`

### Modificação crítica no plan_generator.py

O `SYSTEM_PROMPT` atual define `Exercise` como:
```
{ "name": <string>, "muscle_group": <string>, "sets": <number>, "reps": <string>, "rest_seconds": <number>, "note": <string|null> }
```

Deve ser atualizado para:
```
{ "name": <string>, "muscle_group": <string>, "sets": <number>, "reps": <string>, "rest_seconds": <number>, "note": <string|null>, "home_alternative": { "name": <string>, "muscle_group": <string>, "sets": <number>, "reps": <string>, "rest_seconds": <number>, "note": <string|null> } }
```

Adicionar instrução ao prompt:
```
- home_alternative: versão do mesmo exercício para fazer em casa (apenas peso corporal, cadeira, elástico, garrafa com areia/água). Mesmo foco muscular. Se não existir equivalente razoável, use null.
```

Adicionar após os states existentes:
```typescript
const [workoutMode, setWorkoutMode] = useState<'gym' | 'home'>('gym')

useEffect(() => {
  if (!id) return
  AsyncStorage.getItem(`workout_mode_${id}`).then((val) => {
    if (val === 'home') setWorkoutMode('home')
  })
}, [id])

const changeMode = useCallback(async (mode: 'gym' | 'home') => {
  setWorkoutMode(mode)
  await AsyncStorage.setItem(`workout_mode_${id}`, mode)
}, [id])
```

### Transformação de sessions para home mode

Antes de passar para WorkoutDayCard, processar sessões:

```typescript
function applyWorkoutMode(sessions: WorkoutSession[], mode: 'gym' | 'home'): WorkoutSession[] {
  if (mode === 'gym') return sessions
  return sessions.map(session => ({
    ...session,
    exercises: session.exercises.map(ex =>
      ex.home_alternative ? { ...ex.home_alternative, home_alternative: undefined } : ex
    ),
  }))
}
```

Usar: `const displaySessions = applyWorkoutMode(weeks[selectedWeek].sessions, workoutMode)`

### Compatibilidade retroativa

Exercícios em análises antigas não terão `home_alternative`. O campo é `home_alternative?: HomeExercise | null` (opcional). A função `applyWorkoutMode` usa `ex.home_alternative ?` — se undefined/null, retorna o exercício original. Nunca quebra UI.

Arquivo: `apps/mobile/src/components/workout/WorkoutDayCard.tsx`
Recebe `session: WorkoutSession`. Se a transformação for feita antes de passar, WorkoutDayCard não precisa ser modificado. Preferir a abordagem de processar sessions em workout.tsx.
